# Incremental native list insertion

AiOffice `0.2.0.dev27` can insert a bullet or numbered list into an imported DOCX
without rebuilding existing body content or normalizing the package through the
JSON projection. JSON expresses the requested list and records change evidence;
`/word/document.xml`, `/word/numbering.xml`, their relationships, and OPC content
types remain the native authority.

## Operations

Lists use the same stable-ID structural operations as other body blocks:

```json
{
  "op": "node.insert_after",
  "target": "#recommendation",
  "content": {
    "id": "release_steps",
    "type": "ordered_list",
    "items": [
      "Validate the evidence",
      "Approve the release",
      "Notify stakeholders"
    ]
  }
}
```

`node.insert_before` places the complete list before a mapped top-level anchor.
`node.append` with target `$` places it in the final semantic section, immediately
before an optional terminal body-level `w:sectPr`. All three operations are atomic.

Each item compiles to exactly one `w:p`. Those paragraphs form one contiguous native
range represented by the list's stable ID and a `w:p-group` native reference. The
ID belongs to the list, not to an individual item; per-item editing and nested lists
are deliberately outside this contract.

## Supported subset

The native insertion subset accepts:

- `bullet_list` and `ordered_list`;
- one or more plain-text items;
- a caller-selected list ID or a semantic ID generated by the Patch transaction;
- all root, before-anchor, and after-anchor placements allowed by structural
  editing;
- use of the new list as an anchor, move target, or removal target later in the
  same Patch;
- section-start rebinding when insertion before a later section's first node makes
  the list that section's new semantic start.

Rich runs, hyperlinks, dynamic fields, per-item formatting, multi-level numbering,
custom bullet glyphs, custom starting values, and continuation of an existing
numbering sequence are unsupported. AiOffice rejects unsupported fields instead of
approximating them.

## Numbering isolation

Word stores list behavior outside the paragraph in `/word/numbering.xml`. Reusing a
nearby `numId` could make an inserted numbered list continue an unrelated sequence,
or let a later Word edit couple two lists unexpectedly. AiOffice therefore appends
one fresh single-level `w:abstractNum` and one fresh `w:num` for every inserted list:

- bullet lists use Word's compatible Symbol-font bullet glyph and a bullet number
  format;
- ordered lists use decimal `%1.` formatting starting at 1;
- `abstractNumId` and `numId` are collision-safe decimal IDs;
- each inserted list receives its own `numId`, even beside a list of the same type;
- numbering restarts deterministically for every inserted ordered list;
- all `w:abstractNum` definitions remain before `w:num` instances as required by the
  OOXML numbering schema;
- all existing numbering children remain XML-equivalent and are never renumbered or
  reconstructed.

If the source package has no numbering part, AiOffice creates the canonical
`/word/numbering.xml` part, the internal document relationship, and the required
OPC content-type override. If they already exist, there must be at most one
canonical relationship and one correct override. Unknown numbering children are
preserved.

Removing a newly inserted list does not immediately delete its now-unreferenced
numbering definition. This conservative orphan policy avoids package-wide garbage
collection until AiOffice can prove that no unknown native consumer depends on a
definition.

## Identity and batch semantics

Every inserted item paragraph receives a collision-safe `w14:paraId`. The root
manifest record stores the complete ordered paragraph range, its first paragraph
anchor, indices, and fingerprints. A standalone export and reopen therefore retain
the caller-visible list ID.

Native lowering tracks the actual XML element objects throughout a Patch. A newly
inserted list can immediately:

1. anchor another before/after insertion;
2. move before or after another block as one contiguous unit;
3. be removed as one unit.

After the transaction, AiOffice refreshes shifted content, section, field, table,
cell, and header/footer references and serializes the changed native parts plus the
identity manifest together.

## Section safety

Insertion before a later section's current `start_at` rebinds that section to the
new list while leaving the preceding `w:sectPr` in place. Insertion after a paragraph
that carries `w:sectPr` is refused because it would silently alter section
ownership. Root append requires any body-level `w:sectPr` to be unique and terminal.

The presence of a numbering definition does not imply section ownership. Section
placement is proven from the complete body range and existing section carriers,
independently of `numId`.

## Fail-closed checks

The entire Patch is rejected without mutating the source document when AiOffice
finds:

- a forged or recursively supplied native `source_ref`;
- empty, rich, nested, or otherwise unsupported list content;
- unsafe XML characters in any item;
- a malformed numbering root or non-decimal numbering IDs;
- existing numbering children outside OOXML schema order;
- duplicate `abstractNumId` or `numId` values;
- multiple, external, noncanonical, or conflicting numbering relationships;
- duplicate or incorrect numbering content-type overrides;
- a detached, missing, non-top-level, overlapping, or noncontiguous anchor range;
- an unsafe section boundary or a result that fails identity refresh and semantic
  reprojection.

Existing package bytes remain unchanged on every failed proof.

## CLI, Workspace, and capability discovery

No list-specific binary channel is needed:

```bash
aioffice apply report.docx insert-list.json --output report-updated.docx
aioffice workspace apply ARTIFACT_ID insert-list.json --root project
```

Inspect `Document.capabilities()["structural_editing"]["list_insertion"]` before
planning. It reports supported list types, item content, native group and item
kinds, numbering isolation and restart behavior, missing-part creation, same-Patch
structural operations, and unsupported rich or nested content.

The JSON projection is suitable for model intent, validation, stable selectors, and
change evidence. It is not a lossless serialization of Word numbering. Native
rendering and standalone reopen are the final authority for numbering appearance,
indentation, pagination, and compatibility.
